# Models Guide

## Purpose
`models/` defines SQLModel tables and shared model bases.

## Rules
- Keep models focused on persistence shape and field-level constraints.
- Reuse `models/base_model.py` for the shared UUIDv7 primary key, timestamps, soft-delete fields, and `performed_by`.
- Do not move request parsing, list filtering, or sorting rules into model classes.
- Keep naming aligned with the actual database contract. If a field rename is needed, change model, schema, and migration together.
- Prefer explicit `Field(...)` constraints and indexes in the model rather than burying them in service logic.

## Migration Discipline
- Any model edit that changes the table shape must be followed by `alembic revision --autogenerate ...`.
- Review the autogenerated migration before manual edits.
- If a new model is added, make sure the metadata-loading import path includes it so Alembic and tests see the table.
